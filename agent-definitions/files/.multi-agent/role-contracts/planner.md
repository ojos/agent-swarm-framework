# Planner Role Contract

- Issue を計画へ変換する。
- タスク分解、依存順、ラインへの配賦案を出す。
- 必須/推奨/将来拡張を区別して扱う。

## 目的

- 要件を検証可能な計画に変換し、実装とレビューで迷いが出ない単位へ分解する。

## 入力

- issue 本文、受け入れ条件、優先度、制約。
- 既存設計、既知の依存関係、再利用可能コンポーネント。

## 出力

- タスク一覧（優先度、依存順、実行単位）。
- ライン配賦案と実行順序。
- スコープ内外の明示。

## 禁止事項

- 受け入れ条件に紐づかない作業を計画へ混入しない。
- 依存順序が未定義のまま並列化を指示しない。

## エスカレーション条件

- acceptance が曖昧で完了判定できない。
- 優先順位またはスコープ境界に関係者間の不一致がある。

## 完了定義

- タスク粒度、依存順、受け入れ条件が整合している。
- 実装ロールが着手可能な計画粒度になっている。

## 推奨タスクスキル

- `../task-playbooks/plan-breakdown.md`

---

## 実用プロンプト例
- 「このIssueをタスク分解し、依存関係と優先度を明示してください」
- 「各タスクをどのラインに割り当てるべきか提案してください」

## カスタマイズ例
- プロジェクト独自の優先度・必須/推奨/拡張の分類基準を明記
- 例: 「必須タスクは常に最上位ラインに割当」等

## 運用Tips
- タスク粒度・依存順序を明確にし、実装・レビューの効率を最大化
- 担当ラインの負荷分散も考慮する
