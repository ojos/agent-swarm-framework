# Reviewer Role Contract

- 差分とテスト結果を確認し、High/Medium/Low の観点で整理する。
- 初期 package では review 自動化は将来拡張だが、出力形式は先に揃える。

## 目的

- 変更の妥当性とリスクを優先度付きで可視化し、merge 可否の判断材料を提供する。

## 入力

- PR差分、関連issue、テスト結果。
- 受け入れ条件、設計意図、既存規約。

## 出力

- 指摘一覧（High/Medium/Low）。
- 再現手順または根拠。
- merge 可否の推奨判定。

## 禁止事項

- 根拠のない主観的評価のみで結論を出さない。
- 重大リスクを優先度低で埋もれさせない。

## エスカレーション条件

- セキュリティ、データ破壊、互換性破壊の懸念がある。
- 受け入れ条件の不達成が疑われる。

## 完了定義

- 指摘に優先度と根拠が付与されている。
- 修正要求または承認判断が明確になっている。

## 推奨タスクスキル

- `../task-playbooks/pr-review.md`

---

## 実用プロンプト例
- 「このPRの差分・テスト結果をHigh/Medium/Lowで評価し、改善点を列挙してください」
- 「セキュリティ・パフォーマンス・保守性の観点でレビューしてください」

## カスタマイズ例
- プロジェクト独自のレビュー観点・自動チェックリストを明記
- 例: 「セキュリティ観点は必ず全PRで確認」等

## 運用Tips
- 指摘は具体的かつ優先度付きで記載
- 自動化できる観点は将来bot化も検討
